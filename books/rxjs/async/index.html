# Async code

Follow me on Twitter, happy to take your suggestions on topics or improvements /Chris

Async code is code that isn't done immediately when being called.

setTimeout(() => {
  console.log('do stuff');
}, 3000 )
1
2
3

3 seconds in the future, the timeout is done and do stuff is echoed to the screen. We can see that the anonymous function we provide is being triggered when time has passed. Now, for another more revealing example:

doWork( () => {
  console.log('call me when done');
})
1
2
3
function doWork(cb){
  setTimeout( () => {
    cb();
  }, 3000)
}

input.on('click', () => {

The gist of callbacks and async in general is that one or more methods are invoked sometime in the future, unknown when.

# The Problem

So, we established that a callback can be a timer, ajax code or even an event. But, what is the problem with all that?

One word Readability

Imagine doing the following code

syncCode() // emit 1
syncCode2() // emit 2
asyncCode() // emit 3
syncCode4() // emit 4
1
2
3
4

The output could very well be

1,2,4,3
1

Because the async method may take a long time to finish. There is really no way of knowing, by looking at it, when something finishes. The problem is that if we care about the order so that we get the following output: 1, 2, 3, 4

We might resort to a callback making it look like

syncCode()
syncCode2()
asyncCode(()= > {
  syncCode4()
})
1
2
3
4
5

At this point it is somewhat readable. But, imagine if we have only async code, then it might look like:

asyncCode(() => {
  asyncCode2(() => {
    asyncCode3() => {}
  })
})
1
2
3
4
5

Also known as callback hell, pause for effect 😃

For that reason, promises started to exist so we got code looking like

getData()
  .then(getMoreData)
  .then(getEvenMoreData)
1
2
3

This is great for Request/Response patterns but for more advanced async scenarios I dare say only Rxjs fits the bill.
